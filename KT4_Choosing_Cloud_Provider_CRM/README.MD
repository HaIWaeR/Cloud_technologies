# Анализ инфраструктурных и технических требований CRM системы 

CRM состоит из трех компонентов

1. __Telegram-бот__ (постоянно работает, принимает заказы).
2. __Сайт (Frontend + Backend)__ (API для бота и веб-интерфейс для предпринимателей).
3. __База данных__ (хранит заказы, пользователей, статусы).

Если делаеть MVP проект дял малой аудитории то все три компонента вполне уживутся на одном сервере.


* __CPU: 2 ядра, с частотой от 2.0 ГГц__ -  Достаточно для обработки бота и нескольких пользователей сайта.
* __RAM:__ Нам нужно запустить веб-сервер, бота и базу данных.
    * __ОС:__ 512 МБ
    * __БД:__ 256-512 `МБ чем больше, тем быстрее кеширование, потому лучше взять с запасом.`
    * __Телеграм/Бот:__ 256-512 МБ.

    Итого: Минимум 1 ГБ / 1024 МБ, однако если есть возможность заплатить [258 рублей](#price-calc) поверх то лучше стартовать с 2 ГБ RAM 

* __SSD/NVMe:__ 10-20 ГБ. Сам проект легкий, но нужно место под базу данных. Наш вариант только SSD, дабы избежать заторможеной работы сайта.
* __Сеть:__ Стабильный выход в интернет. Важно, чтобы провайдер имел хорошие каналы для работы с API Telegram (иногда российские провайдеры могут испытывать трудности с соединением к API, поэтому наличие зарубежных серверов у провайдера будет плюсом).

# Расмотрение 5 облачных провайдеров для анализа

## Timeweb Cloud
* Один из крупнейших российских провайдеров который работает с 2006 года

* Дают гранты стартапам что являеться весоммым бонусом, 

__Спектр услуг:__ [VPS/VDS](#vps-kvm), облачные серверы, выделенные серверы, объектное хранилище [S3](#s3), managed [Kubernetes](#kubernetes), хостинг для [1С-Битрикс](#bitrix) 
__Минусы из отзывов:__ Проблемы с почтой (письма могут попадать в спам из-за внутреннего фильтра), поддержка иногда отвечает шаблонно 


## Beget 
* Работает с 2007 года, питерская компания 
* 30 дней бесплатного теста + лучший показатель на рынке

__Плюс для CRM:__ Из коробки поддерживает [Redis](#redis), [Memcached](#memcached), [PostgreSQL](#postgreSQL) что в дальнейшем пригодится для бэкенда, так же уже есть приятный опыт работы с этим сервесом. # Технические требования к серверу


## Telegram-боты работают в одном из двух режимов:
| Режим | Как работает | Нагрузка на сервер |
|-------|--------------|-------------------|
| Long Polling | Бот сам постоянно "спрашивает" Telegram, нет ли новых сообщений (цикл запрос-ответ). Бот всегда висит в памяти. | Постоянная, но небольшая |
| Webhook | Telegram сам стучится к твоему серверу, когда приходит сообщение. Бот "просыпается" только в момент обработки. | Импульсная |

Наш вариант Webhook. Он экономит ресурсы и проще масштабируется. Но требует, чтобы  сервер был доступен из интернета по HTTPS => нужен SSL

## RAM  
* Операционная система (Ubuntu/Debian) Сама ОС и базовые службы	200-300 МБ
* PostgreSQL Буферы, кеши запросов, соединения 256-512 МБ
* Backend (API + веб-кабинет) фреймворки и обработки запросов 150-300 МБ
* Telegram-бот (webhook) Процесс бота + ожидание вызовов от Telegram 50-150 МБ
* Nginx/Apache (веб-сервер)	Обработка HTTP-запросов, раздача статики 50-100 МБ
* Запас на пиковые нагрузки	Когда приходит много заказов одновременно 300-500 МБ

Сумарно по минимуму окло 1 ГБ но учитывая рост числа пользователей, установку Redis для кеширования и пиковые нагрузки вечером оптимальным нашим вариантом будет __2 ГБ__

## CPU 

* PostgreSQL На SELECT/INSERT запросы нагурзка небольшая, но на сложные отчеты будет средняя
* Backend Обработка API запросов, рендер страниц кабинета
* Бот Минимальная

1 ядро справится, но при одновременных запросах от 10+ предпринимателей может тормозить потому так же делаем акцент на 2 ядра для комфортной работы

## SSD/NVMe 

* Операционная система 5-10 ГБ
* Код CRM (бот + бэкенд) 100-500 МБ
* База данных PostgreSQL зависит от числа заказов. 1000 заказов = 10-50 МБ
* Логи (nginx, бота, PostgreSQL, и самого бэка)	1-2 ГБ
* Загрузки пользователей (аватарки, файлы, и прочие) зависит от количетсва пользователей

Мнимумом будет являться 15-20 ГБ, но мы берём с запасом на рост 30-40 ГБ

## <a id="s3"></a> S3-хранилище

__Описание:__ Object Storage (объектное хранилище) - способ хранения файлов в виде "объектов" (картинки, документы, видео, [бэкапы](#backups)). Доступ к файлам идет по HTTP/HTTPS через API. Самый популярный стандарт - Amazon S3. У всех крупных провайдеров есть свои аналоги (VK S3, Timeweb S3, Selectel S3).

__Надо ли:__ ДА, в перспективе. В CRM предприниматели могут загружать файлы (например, фото товаров, чеки, договоры). Хранить их прямо в базе данных это плохая практика ведь база раздувается и тормозит. Лучше складывать в S3, а в БД хранить только ссылку на файл. На старте можно обойтись обычной папкой на сервере, но при росте проекта S3 становится необходим.

__Плюс к проекту:__ удобное масштабирование хранения файлов


## <a id="kubernetes"></a> Kubernetes (K8s)


__Описание:__ Система для автоматического развертывания, масштабирования и управления контейнеризированными приложениями таких как Docker. Позволяет запускать десятки копий приложения, автоматически перезапускать упавшие сервисы, распределять нагрузку.


__Надо ли:__ НЕТ, на старте и даже в среднем проекте. Kubernetes - это мощно, но сложно. Нам потребуется команда DevOps, чтобы его настроить. Для CRM из 3 компонентов (бот, бэкенд, БД) достаточно одного сервера или простого docker-compose. Kubernetes нужен, когда будет 50+ серверов и миллионы пользователей.

__Плюс к проекту:__ -


## <a id="bitrix"></a> Хостинг для 1С-Битрикс

__Описание:__ 1С-Битрикс популярная CMS система для корпоративных сайтов и интернет-магазинов в России. Обычно включает: nginx + php-fpm + MySQL с особыми настройками кеширования, memcached, opcache.

__Надо ли нам:__ НЕТ. поскольку у нас своя CRM с нуля, и мы не использем движок 1С-Битрикс. Эти хостинги заточены под PHP и конкретный движок.

__Плюс к проекту:__ -


## <a id="redis"></a> Redis

__Описание:__ Хранилище данных в оперативной памяти. Работает очень быстро до микросекунды. Используется для:
* Кеширования, чтобы не ходить каждый раз в тяжелую базу данных
* Хранения сессий пользователей
* Брокера сообщений, очереди задач
* Rate limiting - ограничение частоты запросов

__Надо ли нам:__ ДА, желательно. В CRM можно использовать Redis для:
* Кеширования частых запросов например, список заказов предпринимателя
* Хранения сессий, чтобы пользователя не выкидывало при перезапуске бэкенда
* Очередей для отправки уведомлений в Telegram чтобы бот не падал под нагрузкой

__Плюс к проекту:__ + скорость работы, + стабильность при нагрузке


## <a id="memcached"></a> Memcached

__Описание:__ Простое кеш-хранилище в оперативной памяти. Как Redis, но беднее: умеет только хранить ключ-значение, нет продвинутых структур данных, нет сохранения на диск.

__Надо ли нам:__ НЕТ, Memcached проще, но Redis делает всё то же самое и гораздо больше. Memcached имеет смысл, если бы сидел на древней версии PHP но нам нужна максимальная простота без возможности расширения.

__Плюс к проекту:__ Нету поскольку, Redis закрывает все потребности


 
## <a id="postgresql"></a>PostgreSQL

__Описание:__ Мощная объектно-реляционная база данных с открытым исходным кодом. Поддерживает: сложные запросы, транзакции, внешние ключи, JSONB которая позваляет хранить полуструктурированные данные, полнотекстовый поиск, расширения PostGIS для геоданных.

__Надо ли нам:__ ДА, и он уже включён в стек CRM. Он надежный, производительный, не теряет данные в отличие от SQLite при обрыве питания, справляется с тысячами запросов в секунду.

__Плюс к проекту:__ + надежность, + целостность данных, + масштабируемость


## <a id="vps-kvm"></a>VPS / VDS (KVM)

__Описание:__
* VPS/VDS (Virtual Private/Dedicated Server) - виртуальный сервер, который ведет себя как полноценный физический компьютер. Eсть root-доступ, своя ОС, свои ресурсы (CPU, RAM, диск).

* KVM (Kernel-based Virtual Machine) - технология виртуализации на уровне ядра Linux. Самый популярный и "честный" тип виртуализации. KVM дает гарантированные ресурсы - сосед по серверу не может отжать твой процессор или память.

* KVM (отдельно): Это золотой стандарт хостинга. Почти все качественные провайдеры такие как Timeweb, Beget, JustHost, RUVDS используют именно KVM. OpenVZ дешевле, но менее стабилен - лучше не брать.

__Надо ли нам:__ ДА, обязательно KVM. Для CRM нужен стабильный сервер с гарантированными ресурсами. Бот и база данных будут работать 24/7, нельзя чтобы замедлялся сервер. Берём только VPS/KVM.

__Плюс к проекту:__ + стабильность


## <a id="ddos"></a> Защита от DDoS

__Описание:__ DDoS (Distributed Denial of Service) - атака, когда тысячи компьютеров одновременно шлют запросы на сервер, чтобы перегрузить систему и положить сайт или бота. Защита фильтрует плохой трафик, пропуская только реальных пользователей.

* На уровне провайдера (встроенная) - обычно легкая защита на 1-5 Гбит/с
* Платная (например, Qrator, StormWall) - профессиональная защита от мощных атак

__Надо ли нам:__ ДА, но на перспективу. На старте, пока 10-20 пользователей, вероятность быть атакованым очень мала. Но если проект станет популярным, появятся конкуренты или недоброжелатели и в таком случае DDoS-атака возможна. Для этого надо искать провайдера с хотя бы базовой бесплатной защитой обычно до 1-5 Гбит/с.

__Плюс к проекту:__ + защита от вымогателей и конкурентов на поздних этапах жизни проекта 


## <a id="backups"></a> Бэкапы

__Описание:__ Резервное копирование - сохранение копий данных и кода на случай сбоя, ошибкок и случаев когда удалил базу данных случайной командой или взлома.

* Автоматические бэкапы - провайдер сам раз в день/неделю сохраняет снимок сервера
* Ручные бэкапы - ручками настраиваем скрипты, которые сохраняют базу данных и файлы в S3 или на другой сервер

__Надо ли нам:__ ДА, обязательно. База данных с заказами клиентов - самая ценная часть CRM. Ее потеря означает смерть бизнеса. На старте хватит автоматических бэкапов от провайдера. Потом добавим внешние бэкапы например, раз в день скидывать дамп PostgreSQL в S3.

__Плюс к проекту:__ + сохранность данных клиентов, + возможность откатить ошибку


## <a id="ssl"></a> SSL-сертификат

__Описание:__ SSL/TLS - шифрование данных между браузером или Telegram и CRM. Без SSL все данные такие как логины, пароли, заказы и прочие передаются открытым текстом - их может перехватить кто угодно. SSL-сертификат подтверждает, что соединение зашифровано. Бесплатные варианты: Let's Encrypt (выдает сертификат на 3 месяца, автоматически продлевается). Платные: Comodo, Sectigo, GlobalSign (есть гарантии, зеленая строка в браузере).

__Надо ли нам:__ ДА, обязательно и бесплатно. У проекта должен быть HTTPS и WSS если ьудем использовать веб-сокеты. Let's Encrypt бесплатен и поддерживается всеми провайдерами. Настраивается автоматически через Certbot или встроенные инструменты хостинга.

__Плюс к проекту:__ + безопасность данных, + доверие пользователей, + SEO тобеж Google лучше ранжирует HTTPS-сайты


## <a id="tier3"></a> Дата-центры Tier III

__Описание:__ Tier это уровень международной классификации надежности дата-центра. Всего 4 уровня:
* Tier I - базовый, нет резервирования, часто отключается
* Tier II - частичное резервирование
* Tier III - высокая надежность, можно обслуживать узлы без отключения серверов, uptime 99.982% около 1.6 часа простоя в год
* Tier IV - максимальная надежность, допускает любые обслуживания без отключений, uptime 99.995% около 26 минут простоя в год

__Надо ли нам:__
ДА, но не критично. Tier III хорошо для нас подойдёт, дата-центр не вырубится из-за отключения электричества или замены оборудования. Большинство крупных провайдеров (VK, Timeweb, Selectel) работают в дата-центрах Tier III. Для старта сойдет и Tier II, но при выборе провайдера нужно посмотреть, что он декларирует.

__Плюс к проекту:__ + надежность, + меньше риск простоя


## <a id="dbaas"></a>DBaaS (Database as a Service)

__Описание:__ База данных как сервис - не надо устанавливать и настраивать PostgreSQL/MySQL вручную на сервере. Вместо этого провайдер сам разворачивает БД, делает резервное копирование, обновления, репликацию копирование на запасной сервер. Просто строка подключения (хост, порт, логин, пароль) и можно работать с БД через API/драйвер.

__Надо ли нам:__ ДА, это удобно, но дороже. Если брать DBaaS у того же Timeweb или VK, ты платишь за базу данных как за отдельную услугу (и еще за сервер под бота и бэкенд). Суммарно дороже, чем поднять PostgreSQL самому на том же сервере. Плюсы: не надо думать о бэкапах, репликации, отказоустойчивости. Минусы: дороже.

__Плюс к проекту:__ + удобство администрирования, но минус бюджет
